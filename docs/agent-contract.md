# Agent Contract

This document defines the interface between Codex and DUTPilot. DUTPilot is the deterministic local tool. The agent runs it, reads its structured output, and decides what to do next.

## Input Contract

DUTPilot accepts either a config file or explicit CLI arguments.

Required inputs:

- DUT file: the Verilog/SystemVerilog design under test.
- Testbench file: a self-checking testbench.
- Top module: the top testbench module passed to the simulator.
- Simulator: currently `icarus`.
- Optional `dutpilot.yaml`: preferred for repeatable runs.

Example `dutpilot.yaml`:

```yaml
case: adder
dut: adder.v
tb: tb_adder.v
top: tb_adder
sim: icarus
```

Paths in `dutpilot.yaml` are resolved relative to that config file.

## Input Quality Contract

DUTPilot's input is not just a pair of RTL and testbench files. The testbench must satisfy the self-checking contract.

A useful DUTPilot input includes:

- RTL that can be compiled by the selected backend.
- A testbench that applies meaningful stimulus.
- Checkers that compare DUT outputs against expected behavior.
- Explicit `DUTPILOT_FAIL: <reason>` markers on mismatches.
- A `DUTPILOT_PASS` marker only after all intended checks pass.

If the testbench contains no checker, a completed simulation does not show that functional behavior passed. It only shows that the stimulus ran without an explicit failure.

The agent is responsible for generating or improving the testbench quality. DUTPilot is responsible for executing the local tool, parsing markers/logs, and returning structured feedback.

## Command Contract

YAML mode:

```bash
python3 -m dutpilot.cli verify examples/adder/dutpilot.yaml
```

CLI argument mode:

```bash
python3 -m dutpilot.cli verify \
  --dut examples/adder/adder.v \
  --tb examples/adder/tb_adder.v \
  --top tb_adder \
  --sim icarus
```

The command is non-interactive. It writes a run directory and returns success only when the generated report status is passing.

Waveform viewing is separate and should only be used when requested:

```bash
python3 -m dutpilot.cli wave dutpilot_runs/adder
```

## Output Contract

Each run writes:

```text
dutpilot_runs/<case>/
  logs/
    compile.log
    simulate.log
    transcript.log
  reports/
    report.json
    summary.md
  waves/
    wave.vcd
```

Output files:

- `report.json`: machine-readable source of truth for agent decisions.
- `summary.md`: compact human-readable summary.
- `compile.log`: compiler stdout/stderr.
- `simulate.log`: simulator stdout/stderr.
- `transcript.log`: combined compile and simulate log.
- `wave.vcd`: optional waveform generated by the testbench.

## report.json Fields

Important fields:

- `status`: run result. Current v0.1 emits `pass` or `fail`.
- `stage`: decisive stage. Current v0.1 emits `compile` or `simulate`.
- `primary_error`: first actionable error or failure marker, or `null`.
- `errors`: parsed simulator errors, fatal messages, or syntax diagnostics.
- `warnings`: parsed warnings.
- `artifacts`: paths to logs, reports, waveform, and run directory.
- `return_codes`: compile and simulation process return codes.
- `next_action_hint`: short guidance for the next agent action.

Supporting fields include `case`, `simulator`, `top`, `commands`, and `markers`.

## Agent Decision Rules

- If `status=pass`, report that the DUT passed the provided DUTPilot/Icarus testbench.
- If `stage=compile`, inspect `compile.log` and fix syntax, missing files, module names, top module selection, or compile ordering.
- If `stage=simulate`, inspect `primary_error`, `markers.fail`, `simulate.log`, and the relevant DUT/testbench behavior.
- If a future report uses `status=inconclusive`, repair the testbench so it emits an explicit `DUTPILOT_PASS` or `DUTPILOT_FAIL`.
- If a future report uses `stage=timeout`, inspect clocks, resets, waits, and `$finish` reachability.
- If a future report uses `stage=tool_missing`, install or configure the missing simulator before editing RTL.
- After an edit, rerun the same command and reread `report.json`.

## Scope And Limitations

DUTPilot does not equal complete correctness checking. It does not prove the RTL correct and should not be described as fully verifying a design.

A passing report means the DUT passed the provided self-checking DUTPilot/Icarus testbench for the scenarios that testbench covered.
